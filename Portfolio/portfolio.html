렌더링 성능 최적화 — 글로벌 캐싱과 Space-Time Tradeoff
Dictionary Caching Logic
# 매 프레임 이미지 리사이즈 방지 (CPU 부하 및 드로우콜 최소화)
# 전역 캐시 딕셔너리를 활용한 메모리-연산 속도 교환 패턴
note_cache = {}

def get_note_image(size):
    # 해당 사이즈가 없으면 최초 1회만 연산 수행
    if size not in note_cache:
        # smoothscale: 안티앨리어싱 픽셀 보간 포함, 매우 무거운 연산
        note_cache[size] = pygame.transform\
            .smoothscale(NOTE_IMG, (size, size))

    # 이후 프레임부터는 메모리에서 즉시 반환 (Time Complexity: O(1))
    return note_cache[size]
▸ 문제 — 프로파일러로 특정한 78% CPU 점유

롱노트 20개 이상 구간에서 FPS가 60에서 18 미만으로 급락하는 프레임 스파이크가 발생했습니다. 프로파일러를 돌린 결과, 렌더 루프 내 pygame.transform.smoothscale이 매 프레임 반복 호출되며 전체 CPU 연산의 78%를 점유하고 있었습니다.

▸ 해결 — O(N) 연산을 O(1) 메모리 참조로 전환

size를 Key, 변환된 Image 객체를 Value로 하는 전역 해시맵 캐시를 도입했습니다. 최초 1회만 연산하고 이후에는 메모리에서 즉시 참조하도록 전환해 프레임 드랍을 완전히 제거했습니다. 시간 복잡도를 공간 복잡도로 교환하는 Space-Time Tradeoff의 실무적 효과를 직접 수치로 검증했습니다.

PROJECT 02 · CODE REVIEW
데이터 지향(DOD) 설계 — Object Pool과 CPU 캐시 지역성
Object Pooling System (C++)
class BulletPool {
    // 연속된 메모리 공간 확보 → CPU 캐시 지역성(Cache Locality) 극대화
    std::vector<Bullet> pool;

public:
    // resize()로 런타임 힙 재할당 방지 (Pre-allocation)
    BulletPool(int size) { pool.resize(size); }

    Bullet* Acquire() {
        // 비활성 객체를 선형 탐색 (Cache Hit 보장)
        for (auto& b : pool) {
            if (!b.active) {
                b.active = true;
                return &b; // OS 시스템 콜 없이 포인터만 대여
            }
        }
        return nullptr; // 풀 오버플로우 방어
    }

    // 메모리 파괴 없이 논리적 비활성화 플래그만 변경
    void Release(Bullet* b) { b->active = false; }
};
▸ 문제 — 초당 300회 힙 할당으로 인한 프레임 히치

탄막 슈팅의 특성상 초당 수백 번의 new/delete가 반복되면 OS 메모리 요청 시스템 콜 오버헤드가 누적됩니다. 힙 파편화가 가중되며 보스 페이즈에서 16ms 이상의 프레임 히치가 반복 발생했습니다.

▸ 해결 — Pre-allocation과 Cache Locality 확보

게임 시작 시 std::vector에 최대 탄환 수를 일괄 선할당하고 상태 플래그만 토글하는 Object Pool을 설계했습니다. Linked List 대신 연속 메모리 구조를 선택해 CPU의 L1/L2 캐시 라인 프리패치를 활용했으며, 히치 현상이 완전히 사라졌습니다.

Interface & Event-Driven Architecture (C++)
// 데미지 처리 인터페이스 — 개방-폐쇄 원칙(OCP) 달성
void AMyWeapon::ProcessHit(const FHitResult& Hit) {
    // 하드 캐스팅(Cast<T>) 없이 인터페이스 구현 여부만 검사
    if (Hit.GetActor() && Hit.GetActor()->Implements<UDamageableInterface>()) {
        // 무기는 피격 대상의 구체 클래스(Boss, Grunt 등)를 알 필요 없음
        IDamageableInterface::Execute_TakeDamage(Hit.GetActor(), DamageInfo);
    }
}

// 상태 변경 시 델리게이트 브로드캐스트 (Event-Driven)
void UHealthComponent::ApplyDamage(float Damage) {
    CurrentHealth = FMath::Clamp(CurrentHealth - Damage, 0.0f, MaxHealth);

    // UI와 강하게 결합하지 않고 이벤트만 송출
    OnHealthChanged.Broadcast(CurrentHealth, MaxHealth);
}

File Encryption Engine (Python)
# AES-256 CBC 모드 암호화 및 IV 병합 저장 로직
def encrypt_file(file_path, key):
    # 16바이트 난수 IV 생성 (패턴 분석 공격 방어)
    iv = get_random_bytes(AES.block_size)
    cipher = AES.new(key, AES.MODE_CBC, iv)

    with open(file_path, 'rb') as f:
        plaintext = f.read()

    # PKCS7 패딩 적용 후 암호화
    ciphertext = cipher.encrypt(pad(plaintext, AES.block_size))

    # [핵심] 암호문 헤더에 IV를 직렬화하여 병합 저장
    # 복호화 시 앞 16바이트를 슬라이싱하여 IV를 복원
    with open(file_path, 'wb') as f:
        f.write(iv + ciphertext)
▸ 문제 — IV 누락으로 인한 복호화 불가

CBC 모드는 첫 블록 암호화에 IV가 필수입니다. 초기 구현에서 IV를 파일에 저장하지 않고 버리는 버그로 인해 복호화 시 파일이 깨지는 데이터 손상이 반복됐습니다. 파일마다 고유한 난수 IV를 생성하고 암호문 헤더 앞 16바이트에 병합(iv + ciphertext)하여 직렬화하는 구조로 해소했습니다.

▸ 인사이트 — 공격자 시점에서 얻은 서버 보안 설계

클라이언트 파일 시스템을 직접 제어하고 암호화하는 프로그램을 직접 구현하면서, 역으로 데이터 위변조를 어떻게 방어할 것인지에 대한 구체적 해답을 얻었습니다. 멀티플레이어 서버에서 클라이언트가 보내는 모든 데이터를 Zero Trust 원칙 하에 다뤄야 하며, 무결성 검증과 암호화가 왜 필수적인지 실무 관점으로 이해했습니다.
