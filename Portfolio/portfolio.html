    # 캐시에 해당 사이즈의 이미지가 없다면 최초 1회만 연산 수행

    # 이후 프레임부터는 메모리에 적재된 이미지를 즉시 반환 (Time Complexity: O(1))

PROJECT 02 · CODE REVIEW
데이터 지향(DOD) 기반의 메모리 구조 설계와 캐시 지역성
Object Pooling System (C++)
class BulletPool {
    // 연속된 메모리 공간 확보를 통해 CPU 캐시 지역성(Cache Locality) 극대화
    std::vector<Bullet> pool;
    
public:
    // 풀 초기화 시 resize()를 통해 런타임 중 힙 메모리 재할당 방지
    BulletPool(int size) { pool.resize(size); }

    Bullet* Acquire() {
        // 배열을 순회하며 비활성 객체를 선형 탐색 (Cache Hit 보장)
        for (auto& b : pool) {
            if (!b.active) {
                b.active = true;
                return &b; // 시스템 콜(OS 할당) 없이 포인터만 대여
            }
        }
        return nullptr; // 풀 오버플로우 시 런타임 에러를 막는 방어적 반환
    }
    
    // 객체 메모리를 파괴하지 않고 논리적 비활성화 플래그만 변경
    void Release(Bullet* b) { b->active = false; }
};
▸ 객체 생성 비용의 한계와 힙 파편화(Fragmentation) 극복

탄막 슈팅 게임의 특성상 초당 수백 번의 new 연산과 delete 연산을 수행하면, 매번 운영체제에 메모리를 요청하는 시스템 콜(System Call) 오버헤드가 발생합니다. 동적 할당과 해제가 반복되면 힙 메모리 공간이 조각나는 파편화 현상이 누적되어, 가비지 컬렉터가 없는 순수 C++ 환경에서는 결국 치명적인 성능 저하와 렉을 유발합니다. 이를 원천 차단하기 위해 게임 시작 전 로딩 단계에서 std::vector에 최대 총알 개수만큼 메모리를 일괄 할당(Pre-allocation)해두고, 상태 플래그만 토글하는 오브젝트 풀(Object Pool) 시스템을 설계했습니다.

▸ CPU 캐시 지역성(Cache Locality)의 절대적 우위 확인

Linked List가 아닌 연속된 메모리 블록인 Vector 배열 구조를 채택한 핵심 이유는 하드웨어의 CPU 캐시 지역성 때문입니다. 물리적인 메모리 주소가 조밀하게 연속되어 있어, CPU가 메모리에서 데이터를 가져올 때 주변 데이터까지 한 번에 L1/L2 캐시 라인으로 프리패치(Prefetch)하여 Cache Hit 율을 극대화할 수 있었습니다. 이를 통해 데이터 지향 설계(DOD)가 퍼포먼스에 미치는 영향을 직접 체감했습니다.

void UHealthComponent::ApplyDamage(float Damage) {
    CurrentHealth = FMath::Clamp(CurrentHealth - Damage, 0.0f, MaxHealth);
    
    // UI나 다른 시스템과 강하게 결합하지 않고 이벤트만 송출 (Event-Driven)
    OnHealthChanged.Broadcast(CurrentHealth, MaxHealth);
}
▸ 인터페이스를 통한 완벽한 디커플링 (Decoupling)

무기 객체가 타격 대상(일반 몹, 보스, 폭발물 등)의 구체적인 클래스 타입(Concrete Class)을 헤더 파일에 Include하지 않도록 UDamageableInterface를 도입했습니다. 무기는 오직 '데미지를 가한다'는 다형성 메시지만 전송하며, 타격받는 액터가 각자의 체력, 방어력, 무적 상태 로직에 따라 자율적으로 데미지를 연산합니다. 새로운 형태의 적이나 파괴 가능한 환경 오브젝트를 백 개 이상 추가하더라도 무기 코드를 단 한 줄도 수정할 필요가 없는 완벽한 의존성 역전 구조를 구현했습니다.

▸ 하드 레퍼런싱 방지와 메모리 연쇄 로드 차단

초기 버전에서는 체력 UI를 갱신하기 위해 캐릭터 클래스가 UMG UI 클래스를 직접 캐스팅(Cast)하여 참조했습니다. 이러한 강한 결합(Hard Reference)은 언리얼 엔진이 캐릭터를 메모리에 스폰할 때, 의존성 트리에 묶인 불필요한 UI 텍스처와 사운드 애셋들까지 연쇄적으로 강제 로드하게 만드는 치명적인 원인이 됩니다. 이를 해결하고자 언리얼의 멀티캐스트 델리게이트(Multicast Delegate)를 활용한 이벤트 주도 아키텍처로 전면 리팩토링했습니다. 뷰(UI)는 모델(Character)의 이벤트를 수동적으로 구독만 하게 되어, 메모리 풋프린트를 크게 줄이고 레벨 로딩 속도를 최적화했습니다.
