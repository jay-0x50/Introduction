PROJECT 02 · CODE REVIEW
데이터 지향(DOD) 기반의 메모리 구조 설계와 캐시 지역성
Object Pooling System (C++)
class BulletPool {
    // 연속된 메모리 공간 확보를 통해 CPU 캐시 지역성(Cache Locality) 극대화
    std::vector<Bullet> pool;
    
public:
    // 풀 초기화 시 resize()를 통해 런타임 중 힙 메모리 재할당 방지
    BulletPool(int size) { pool.resize(size); }

    Bullet* Acquire() {
        // 배열을 순회하며 비활성 객체를 선형 탐색 (Cache Hit 보장)
        for (auto& b : pool) {
            if (!b.active) {
                b.active = true;
                return &b; // 시스템 콜(OS 할당) 없이 포인터만 대여
            }
        }
        return nullptr; // 풀 오버플로우 시 런타임 에러를 막는 방어적 반환
    }
    
    // 객체 메모리를 파괴하지 않고 논리적 비활성화 플래그만 변경
    void Release(Bullet* b) { b->active = false; }
};
▸ 객체 생성 비용의 한계와 힙 파편화(Fragmentation) 극복

탄막 슈팅 게임의 특성상 초당 수백 번의 new 연산과 delete 연산을 수행하면, 매번 운영체제에 메모리를 요청하는 시스템 콜(System Call) 오버헤드가 발생합니다. 동적 할당과 해제가 반복되면 힙 메모리 공간이 조각나는 파편화 현상이 누적되어, 가비지 컬렉터가 없는 순수 C++ 환경에서는 결국 치명적인 성능 저하와 렉을 유발합니다. 이를 원천 차단하기 위해 게임 시작 전 로딩 단계에서 std::vector에 최대 총알 개수만큼 메모리를 일괄 할당(Pre-allocation)해두고, 상태 플래그만 토글하는 오브젝트 풀(Object Pool) 시스템을 설계했습니다.

▸ CPU 캐시 지역성(Cache Locality)의 절대적 우위 확인

Linked List가 아닌 연속된 메모리 블록인 Vector 배열 구조를 채택한 핵심 이유는 하드웨어의 CPU 캐시 지역성 때문입니다. 물리적인 메모리 주소가 조밀하게 연속되어 있어, CPU가 메모리에서 데이터를 가져올 때 주변 데이터까지 한 번에 L1/L2 캐시 라인으로 프리패치(Prefetch)하여 Cache Hit 율을 극대화할 수 있었습니다. 이를 통해 데이터 지향 설계(DOD)가 퍼포먼스에 미치는 영향을 직접 체감했습니다.

블록 암호화 패딩 및 IV 생명주기 관리
AES-256 Encryption & Padding (Python)
from Crypto.Cipher import AES
from Crypto.Util.Padding import pad, unpad
from Crypto.Random import get_random_bytes

def encrypt_file(file_path, key):
    # 1. 파일마다 고유한 16바이트 초기화 벡터(IV) 난수 생성
    iv = get_random_bytes(AES.block_size)
    cipher = AES.new(key, AES.MODE_CBC, iv)
    
    with open(file_path, 'rb') as f:
        plaintext = f.read()
        
    # 2. PKCS7 패딩: 데이터 크기를 16바이트 배수로 정확히 맞춤
    padded_data = pad(plaintext, AES.block_size)
    ciphertext = cipher.encrypt(padded_data)
    
    with open(file_path, 'wb') as f:
        # 3. 복호화를 위해 암호문 맨 앞에 IV 결합 저장
        f.write(iv + ciphertext)
▸ 대칭키 암호화 알고리즘의 한계와 블록 패딩(Padding)의 필수성

AES 알고리즘은 데이터를 16바이트(128비트) 블록 단위로 쪼개어 연산하는 블록 암호입니다. 만약 원본 파일의 크기가 16바이트로 나누어 떨어지지 않으면 암호화 과정에서 즉시 알고리즘 오류가 발생합니다. 이를 방지하기 위해 데이터의 남은 공간을 특정한 바이트 값으로 채우는 PKCS7 패딩 기법을 적용했습니다. 이 과정에서 메모리 버퍼 상의 데이터 구조를 정확하게 맞추지 않으면 파일이 영구적으로 파손된다는 점을 경험하며, 로우 레벨 데이터 바이트 스트림 조작의 중요성을 체감했습니다.

▸ CBC 모드의 보안 취약점 극복과 IV 은닉 기법

단순한 ECB 모드로 암호화를 진행하면 동일한 평문 블록이 항상 동일한 암호문 블록으로 변환되어 패턴 공격에 극히 취약해집니다. 이를 막기 위해 앞 블록의 암호문이 다음 블록 연산에 영향을 주는 CBC(Cipher Block Chaining) 모드를 적용했습니다. 이때 연산의 시작점이 되는 IV(초기화 벡터)를 고정된 값으로 사용하면 보안이 무너지므로, 파일마다 get_random_bytes를 통해 고유한 난수 IV를 생성했습니다. 생성된 IV는 암호문 맨 앞 16바이트에 헤더처럼 부착하여 저장함으로써, 복호화 시 키만 있다면 IV를 분리하여 원본 데이터를 안전하게 복구해내는 데이터 직렬화/역직렬화 구조를 성공적으로 설계했습니다.
